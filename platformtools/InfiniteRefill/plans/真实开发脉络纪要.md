## 7. 单次续杯升级：探测(401/429) -> 上报 -> topup 下发账号 JSON -> 写盘 -> 删除失效文件

用户在后期明确了“单次续杯”需要升级为完整流水线：

1) 逐个探测本地账号文件（状态码以 `wham/usage` 响应为准）
2) 触发规则：
   - 一组账号里 `401 + 429 >= 8` 触发
   - 或本地数量不足目标配额时触发（bootstrap）
3) 触发后行为：
   - 删除本地 `status_code ∈ {401,429}` 对应文件（删除前备份）
   - 向服务端发送每个账号的状态（401=失效；429=配额用光，服务端可据此“一周冷却再放出”）
4) 服务端下发账号 JSON（topup 响应），客户端写入 `auth-dir`

接口契约（客户端侧与文档侧已固定，服务端已实现）：

- `POST /v1/refill/topup`：返回 `accounts[]`，每项含 `{file_name, auth_json}`（详见 [`客户端/README.md`](../客户端/README.md:58) 的说明口径）。
- 实现入口：[`server.index.fetch()`](../server/src/index.ts:2053)

落地脚本：

- [`客户端/普通用户/单次续杯.bat`](../客户端/普通用户/单次续杯.bat:1)

## 10. 已知工具/编辑踩坑（本次对话真实发生）

## 11. 后期新增：服务端生成平台密钥 + 批量分发包（离线分发链路）

用户在后期新增并确认：平台密钥（`USER_KEY` / `UPLOAD_KEY`）应由服务端生成（管理员申请批量生成），而不是管理员手工导入明文；并需要支持“一次申请 N 个密钥，返回 N 个**标准 zip 密码压缩包**”，用于线下/离线手动分发。

关键规则（写入接口契约与实现）：

- 每个压缩包命名：`1.zip .. N.zip`（数字命名）。
- zip 密码 = 该包对应的密钥（用户用 7zip/系统解压，输入密钥即可打开）。
- 服务端同时生成 `分发清单.txt`：每行 `压缩包名：对应密钥`。
- 服务端允许返回下载链接（写入 R2，并返回 presigned GET URL）。

代码落地接口：[`server.index.fetch()`](../server/src/index.ts:429) → `POST /admin/packages/issue`（入口见 [`server/src/index.ts`](../server/src/index.ts:540)）。

## 17. 下一步（让线上 /admin/packages/issue 真正可用）

在不改动/不删除你本地已保存的密钥文件的前提下，线上要跑通必须把 R2 S3 API token 注入 Worker secrets：

- `cd server && npx wrangler secret put R2_ACCESS_KEY_ID`
- `cd server && npx wrangler secret put R2_SECRET_ACCESS_KEY`

注入后执行一次发布：

- `cd server && npx wrangler deploy`

完成后，`POST /admin/packages/issue` 将不再触发 `missing_r2_s3_credentials` 分支，并会返回每个 zip 的 presigned 下载链接与清单下载链接（前提：R2 bucket 绑定与 `R2_ACCOUNT_ID/R2_BUCKET_NAME` 正确）。

## 19. 真实发生的安全事件与处置（必须写入纪要）

### 19.1 用户在聊天中粘贴过 Cloudflare API Token

处置原则：不复述、不存储、不使用，要求立刻撤销并重新签发。

并明确区分：Cloudflare API Token **不是** R2 S3 API token（`R2_ACCESS_KEY_ID` / `R2_SECRET_ACCESS_KEY`），后者用于 SigV4 presigned URL（实现见 [`server.index.fetch()`](../server/src/index.ts:540) 中的 [`AwsClient`](../server/src/index.ts:576)）。

### 19.2 用户坚持“本地明文保存密钥”和“不删除本地密钥文件”

对仓库与协作的硬约束被写死为：

- 真实密钥不可进入可提交文件与聊天记录（见本纪要第 16 节）。
- 用户本地明文保存可以存在，但必须在：
  - Cloudflare secrets（`wrangler secret put`），或
  - 被 gitignore 的本地文件（例如 `server/.dev.vars`，示例仅供参考：[`server/.dev.vars.example`](../server/.dev.vars.example:1)）。

为了保护用户“本地已保存的值不被后续提交/操作影响”，对话中使用过 git 的跳过追踪策略（skip-worktree）：

- `git update-index --skip-worktree server/wrangler.toml`

对应文件：[`server/wrangler.toml`](../server/wrangler.toml:1)。

### 19.3 示例文件曾被填入真实密钥 → 已恢复占位符 + 仓库扫描

对话中曾出现“把真实 R2 密钥写进示例文件”的情况；最终处理策略是：

- 示例文件必须回到占位符（例如 `REPLACE_ME`），只作为字段名/用途参考。
- 工作区与提交内容需做全文搜索，确保敏感值不残留。

对应示例文件：[`server/.dev.vars.example`](../server/.dev.vars.example:1)。

## 21. 自定义域名绑定：从 routes/zone 到 DNS/TLS 的真实踩坑

本次对话中，自定义域名绑定经历了多次“工具不可用/配置错误/DNS 错解析”导致的失败与修复，最终形成可复用的排障链路。

### 21.1 wrangler 不负责 zones/DNS：`wrangler zones list` 并不存在

- 现象：尝试通过 wrangler 枚举 zone / 直接操作 DNS，得到 `Unknown arguments: zones, list`。
- 结论：wrangler v3 不提供 zones/DNS 管理命令；DNS 必须在 Cloudflare Dashboard 里配置。

### 21.2 route zone 不存在：`code: 10083`

- 现象：首次尝试绑定到错误的 zone（例如把 pattern 当成 zone、或 zone 拼写错误），deploy 失败：`The zone "..." does not exist on your account. [code: 10083]`。
- 修复：确认正确的 Zone 为 `aiaimimi.com`，并将 routes 固定为：
  - [`server/wrangler.toml`](../server/wrangler.toml:5) 中 `routes = [{ pattern = "refill.aiaimimi.com/*", zone_name = "aiaimimi.com" }]`

### 21.3 TLS 失败根因：DNS 解析到保留网段（198.18.0.0/15）

- 现象：Windows `curl` 报 TLS handshake 失败（SChannel），并且对 `https://refill.aiaimimi.com/health` 无法连通。
- 诊断：发现 `refill.aiaimimi.com` 解析到了 `198.18.0.80`（保留网段，RFC 2544），导致真实链路不可达。
- 修复：将 DNS 设置为：
  - `CNAME refill -> infinite-refill-server.<account>.workers.dev`，并开启橙云代理。
- 结论：**workers.dev 仅作为 DNS target**，用户访问依然是自定义域名；Cloudflare 代理层会完成到 Worker 的路由。

### 21.4 代理变量干扰诊断

- 现象：在排查 HTTP/TLS 时，`curl` 可能受 `https_proxy`/`HTTPS_PROXY` 影响，导致“看起来像服务器问题”。
- 策略：验证自定义域名时，明确在终端里清空代理环境变量后重试。

## 24. 生命周期状态机：过期/失效/温养/二次校验（从旧逻辑迁移）

“旧有的过期/失效/二次校验逻辑”被迁移到艺术品生命周期，核心语义如下：

- `available`：公共池可领取
- `claimed`：私有池已绑定领取人
- `quarantine`：局部损坏后温养期（7 天）；到期回公共池
- `deleted`：完全损坏；记录原因与删除时间；绑定关系中断

损坏上报规则：

- 完全损坏（full）：上报后换新；原作品进入 `deleted` 并记录唯一编号；绑定关系中断。
- 局部损坏（partial）：上报后换新；原作品进入 `quarantine`，`eligible_after = now + 7d`；到期回公共池；绑定关系中断。
- 二次校验：服务端后续可做校验；若暂时无法校验，则**先信任客户端**并写入待校验队列。

数据层已落地表（schema + migration）：
