# 平台密钥生成 + 手动分发包 + 有效数据备份恢复 + 防撞库（计划）

对齐背景：

- 你要求平台侧密钥（`USER_KEY` / `UPLOAD_KEY`）必须由服务端生成；超级管理员不再“手工导入明文 key”为主流程。
- 你需要一个“手动分发”链路：管理员一次申请 N 个密钥后，得到 N 个加密压缩包 + 一个映射清单文本，用于离线转交给特殊用户。
- 你担心“唯一 key 会被撞库一直试”。你已确认策略：优先 Cloudflare WAF/Rate Limiting；/admin/* 再加 IP 白名单（Zero Trust Access）；Worker 只做最小兜底。

本计划仅分发平台密钥与客户端配置包；不涉及任何第三方 token/凭据的打包与分发（边界已写入 [`server/DESIGN.md`](../server/DESIGN.md:183)）。

---

## 1) 核心接口契约（建议）

### 1.1 生成平台密钥（管理员）

- `POST /admin/keys/issue`

请求体：

```json
{
  "type": "user",
  "count": 30,
  "label": "2026-02 special",
  "bind_pool_size": 10
}
```

说明：

- `type=user|upload`：生成对应角色的平台密钥。
- `bind_pool_size=10`：仅用于平台策略/配额预留（等价于“这个 key 默认绑定 10 槽位/目标池”），不代表服务端会把第三方凭据打包下发。
- 返回的明文 key 只展示一次；服务端只存 hash。

### 1.2 有效数据备份（管理员）

- `GET /admin/backup/export`

导出内容（仅有效数据）：

- `upload_keys`：仅 enabled=1
- `user_keys`：仅 enabled=1
- `refill_keys`：排除 revoked（或只导出 available/claimed）
- `accounts`：仅 invalid=0，且不包含任何 token 字段（不导出 `access_token/refresh_token/auth_json`）

不导出：

- `probes` 等日志
- `invalid_accounts/exhausted_accounts/pending_verification_accounts` 等“无效库/暂存库”

### 1.3 有效数据恢复（管理员）

- `POST /admin/backup/import`

请求体：

```json
{ "dump": { "version": 1, ... } }
```

恢复策略：

- keys 采用 `INSERT OR IGNORE`，避免覆盖线上现有 key。
- accounts 以 `email_hash` 去重 upsert，仅恢复基础识别信息与 last 状态（不恢复 token 字段）。

## 2) “手动分发包”完整链路（推荐落地为离线打包）

你想要的交付物：

- N 个压缩包（加密/带密码），包名为数字：`1.zip`、`2.zip`...
- 一个文本映射文件 `分发清单.txt`：每行 `压缩包名：对应密钥`
- 压缩包密码 = 对应平台密钥本身

为降低 Worker 端复杂度与二进制处理风险，建议采用：

1) 管理员调用 [`server.index /admin/keys/issue`](../server/src/index.ts:1) 得到 keys（只出现一次）。
2) 在管理员本地运行“打包脚本”（Windows/macOS/Linux）：
   - 输入：服务器地址 + 一组 keys
   - 输出：数字命名 zip + `分发清单.txt`
   - zip 内容建议只含：
     - `无限续杯配置.env`（写入 `SERVER_URL` + `USER_KEY` 或 `UPLOAD_KEY`）
     - `README.txt`（指向 [`客户端/README.md`](../客户端/README.md:1)）

> 这样满足“可离线分发”，同时避免 Worker 侧引入 zip AES 加密与多文件打包。

如果你坚持“服务端直接返回压缩包”，建议改成：

- Worker 生成包后写入 R2
- `POST /admin/keys/export-packages` 返回批次下载链接（或对象列表）

## 3) 防撞库（Cloudflare WAF/Rate Limiting 为主）

### 3.1 WAF/Rate Limiting（建议规则）

对 `/v1/*`：

- 基础限速：按 IP + 路径限制，例如 60 req/min。
- 更严限速：对连续出现 401/403 的 IP（或 bot score 低的请求）限速更严，例如 10 req/min，持续 10~30 分钟。

对 `/admin/*`：

- Cloudflare Zero Trust Access / IP 白名单：只允许固定出口 IP 或通过 Access 鉴权。
- 单独限速：例如 20 req/min。

### 3.2 Worker 最小兜底（不与 WAF 重复造轮子）

- key 格式预校验：只接受 `k_` 前缀 + 固定长度 base64url（减少无意义撞库）。
- 统一错误体：仍按你确认的语义保留 `missing=401`、`invalid=403`，但返回体尽量一致（减少 oracle）。
- （可选）对异常高频的 IP 进行短期拒绝（更推荐后续用 Durable Objects/KV 做短期计数，而不是 D1）。

## 4) 管理员能力重整（客户端脚本层）

将“超级管理员能力”从“导入明文 key”转为“服务端生成 + 备份恢复 + 统计”。

建议文档与客户端入口以如下为主：

- 生成用户密钥：[`客户端/超级管理员/生成用户密钥_申请.bat`](../客户端/超级管理员/生成用户密钥_申请.bat:1)
- 生成上传密钥：[`客户端/超级管理员/生成上传密钥_申请.bat`](../客户端/超级管理员/生成上传密钥_申请.bat:1)
- 导出有效数据备份：[`客户端/超级管理员/导出有效数据_备份.bat`](../客户端/超级管理员/导出有效数据_备份.bat:1)
- 导入有效数据恢复：[`客户端/超级管理员/导入有效数据_恢复.bat`](../客户端/超级管理员/导入有效数据_恢复.bat:1)

旧的“导入 key”接口与脚本仅作为兼容保留（例如迁移老系统时用），默认不推荐。

## 5) 验收标准（不跑第三方 token）

- `POST /admin/keys/issue`：生成的 key 随机性足够、不可预测；服务端仅存 hash；重复调用不会泄露历史明文。
- `GET /admin/backup/export`：输出 dump 仅包含有效数据，且不含任何 token 字段与 probes 日志。
- `POST /admin/backup/import`：可在空库恢复；重复导入不破坏数据（幂等/近似幂等）。
- WAF/Rate Limiting：对 `/v1/*` 的爆破在边缘被限速；`/admin/*` 只有白名单来源可访问。
