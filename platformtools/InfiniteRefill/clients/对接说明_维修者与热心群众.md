# 对接说明：维修者（Repairer）与热心群众（Uploader）

本文用于和外部协作者对接（维修者/热心群众）。以服务端实际实现为准：[`server/src/index.ts`](server/src/index.ts:777)

- **维修者**与**热心群众**使用同一套凭据：`UPLOAD_KEY`（Header: `X-Upload-Key`）。
- 权限继承：`UPLOAD_KEY` ⊇ `USER_KEY`（上传者也可以调用普通用户接口）。见：[`server.index.requireAtLeastUser()`](server/src/index.ts:573)

## 1. 热心群众（Uploader）职责与接口

热心群众主要做两件事：

1) **把“账号观测结果/注册信息”上报到服务端**（帮助平台维护账号库）。
2) （可选）积累投稿积分后，生成普通用户分发包。

### 1.1 上报探测结果：`POST /v1/probe-report`

用途：上报对账号 `wham/usage` 探测得到的 `status_code`（例如 401/429），服务端用于维护：
- `accounts` 聚合表
- `invalid_accounts`（401）
- `exhausted_accounts`（429，冷却 7 天）

接口文档：[`server/API.md`](server/API.md:429)（章节 “POST /v1/probe-report”）

**请求体**（示例）：

```json
{
  "reports": [
    {
      "email_hash": "<sha256_hex>",
      "account_id": "acc_123",
      "status_code": 401,
      "probed_at": "2026-02-28T17:19:14Z"
    }
  ]
}
```

**curl 示例**：

```bash
curl -sS -X POST "$SERVER_URL/v1/probe-report" \
  -H "X-Upload-Key: $UPLOAD_KEY" \
  -H "Content-Type: application/json" \
  -d '{"reports":[{"email_hash":"...","account_id":"acc_123","status_code":401,"probed_at":"2026-02-28T17:19:14Z"}]}'
```

### 1.2 注册账号（可选包含 auth_json）：`POST /v1/accounts/register`

用途：把账号的 `email_hash/account_id/seen_at` 注册到服务端；
可选上传 `auth_json`（敏感）用于后续 `topup` 下发账号文件。

接口文档：[`server/API.md`](server/API.md:431)（章节 “POST /v1/accounts/register”）

**请求体**（示例）：

```json
{
  "accounts": [
    {
      "email_hash": "<sha256_hex>",
      "account_id": "acc_123",
      "seen_at": "2026-02-28T00:00:00Z",
      "auth_json": {
        "type": "codex",
        "access_token": "..."
      }
    }
  ]
}
```

#### 重要：敏感字段与加密存储

- `auth_json` 会被服务端用 `ACCOUNTS_MASTER_KEY_B64` 做 AES-GCM 加密后，存入 D1（`accounts.auth_json`）。实现：[`server.index.accountsAuthJsonEncrypt()`](server/src/index.ts:501)
- 管理员导出接口 **不会导出 token**（不会泄露）。实现：[`server.index.fetch()`](server/src/index.ts:1070)

#### “提交JSON.sh” 的合规检查

脚本：[`客户端/热心群众/提交JSON.sh`](客户端/热心群众/提交JSON.sh:1)

默认会拒绝包含以下字段的 JSON（避免误上传）：
- `access_token`
- `refresh_token`
- `id_token`

如果你需要上传 `auth_json` 做 topup 下发，请按对接口径使用 `POST /v1/accounts/register`，并且确认 payload 的敏感字段策略与你们的协作方式一致。

> 注：该脚本的“阻止敏感字段”属于安全兜底；真正是否允许上传 token，以项目负责人当前策略为准。

### 1.3 生成普通用户分发包（Uploader 积分）：`POST /v1/uploader/packages/issue`

用途：消耗 uploader 的 `distribution_credits`，生成 `USER_KEY` 分发包（zip + 清单，presigned URL）。

接口文档：[`server/API.md`](server/API.md:359)

## 2. 维修者（Repairer）职责与接口

维修者的目标：处理被用户标记为“完全损坏”的作品。

### 2.1 维修区如何产生任务？

普通用户/上传者对自己已 claim 的作品调用：`POST /v1/artworks/report-damage`，并传 `kind=full`。

服务端会把旧作品状态置为 `repair`（进入维修区），并给调用方“交换领取”一个 replacement。

接口文档：[`server/API.md`](server/API.md:266)（章节 “POST /v1/artworks/report-damage”）

### 2.2 领取维修任务：`POST /v1/repairs/claim`

- 从 `repair` 随机领取，状态变更：`repair` → `repair_claimed`
- 返回作品正文

接口文档：[`server/API.md`](server/API.md:293)

**请求体**：

```json
{ "count": 1 }
```

**curl 示例**：

```bash
curl -sS -X POST "$SERVER_URL/v1/repairs/claim" \
  -H "X-Upload-Key: $UPLOAD_KEY" \
  -H "Content-Type: application/json" \
  -d '{"count":1}'
```

### 2.3 提交修复成功：`POST /v1/repairs/submit-fixed`

- 覆盖写回同一个 `artwork_id` 的 R2 对象
- 状态恢复为 `available`

**关键约束**：`fixed_artwork.account_id` 必须存在且等于 `artwork_id`（防止错修/串号）。
实现校验：[`server.index.fetch()`](server/src/index.ts:1719)

**请求体**（示例）：

```json
{
  "artwork_id": "acc_123",
  "fixed_artwork": {
    "account_id": "acc_123",
    "title": "..."
  }
}
```

### 2.4 提交修复失败：`POST /v1/repairs/submit-failed`

- `repair_fail_count += 1`
- 未达 3 次：状态回到 `repair`（可再次被领取）
- 达到 3 次：进入 `graveyard`，并写入 `artwork_tombstones`（禁止以后重复 submit 同 id）

接口文档：[`server/API.md`](server/API.md:326)

### 2.5 误报恢复：`POST /v1/repairs/submit-misreport`

- 把 `repair_claimed` 恢复成 `available`。

接口文档：[`server/API.md`](server/API.md:345)

### 2.6 端到端冒烟（可复现）

仓库提供 E2E 冒烟脚本：[`客户端/维修者/冒烟_维修流程端到端.sh`](客户端/维修者/冒烟_维修流程端到端.sh:1)

覆盖：
- submit(account_id) -> claim -> report-damage(full) -> repairs/claim -> submit-failed×3 -> graveyard+tombstone -> re-submit 被拒绝

## 3. 常见错误码（对接时高频）

以 JSON 响应 `{ ok:false, error:"..." }` 为主：

- `401 missing X-Upload-Key`：没带 header
- `403 invalid upload key`：key 不存在/被禁用
- `409 no_repairable_artwork`：维修区暂时没货
- `409 no_available_artwork`：公共池没作品可领取
- `413 too_many_items / item_too_large`：数量/体积超限

详见：[`server/API.md`](server/API.md:489)
